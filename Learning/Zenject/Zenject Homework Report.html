Отчёт по Dependency Injection и Zenject в проекте SlimeRush

Unity-проект: SlimeRush
Тема: Dependency Injection, Zenject, SceneContext и ProjectContext

1. Цель работы

Цель работы - на практике освоить Dependency Injection в Unity с использованием Zenject, а также разобраться, как организовывать зависимости проекта через installers, SceneContext и ProjectContext.

В проекте SlimeRush была начата архитектурная переработка игровых систем. Прямые зависимости между UI, логикой толпы и расчётом формации были заменены на зависимости от интерфейсов. Это позволило уменьшить связность классов и сделать код более расширяемым.

Важно: на текущем этапе в проекте уже используется Zenject через SceneContext. Полноценный ProjectContext с глобальными сервисами ещё необходимо добавить отдельным шагом, чтобы полностью соответствовать формулировке задания.

2. Интеграция Zenject

В проект был добавлен Zenject. На сцене GameplayPrototype используется SceneContext, к которому подключён GameplayInstaller. Через installer регистрируются зависимости, необходимые для игрового уровня. 

Container.BindInstance(crowdFormationSettings).AsSingle();
Container.Bind<ICrowdFormation>().To<EllipseCrowdFormation>().AsSingle();

Container.Bind<SlimeCrowdManager>().FromInstance(slimeCrowdManager).AsSingle();
Container.Bind<ISlimeCrowd>().FromInstance(slimeCrowdManager).AsSingle();
Container.Bind<ISlimeCrowdCommands>().FromInstance(slimeCrowdManager).AsSingle();

Таким образом, зависимости создаются и связываются централизованно через контейнер Zenject, а не вручную внутри игровых классов.

3. Использование интерфейсов

Для системы слаймов были выделены отдельные интерфейсы: один для чтения состояния толпы, второй для команд, изменяющих состояние.

Интерфейс чтения

public interface ISlimeCrowd
{
    int SlimeCount { get; }
    event Action<int> OnSlimeCountChanged;
}

Командный интерфейс

public interface ISlimeCrowdCommands
{
    void AddSlimes(int amount);
    void RemoveSlimes(int amount);
    void MultiplySlimes(int multiplier);
}

Такое разделение соответствует Interface Segregation Principle. UI получает только ISlimeCrowd и не может изменять количество слаймов. Игровые объекты, например ворота, бонусы или враги, смогут использовать ISlimeCrowdCommands.

4. Вынесение расчёта формации

До рефакторинга SlimeCrowdManager отвечал не только за хранение и создание слаймов, но и за математический расчёт их расположения. Это нарушало принцип единственной ответственности.

Для расчёта формации был создан интерфейс:

public interface ICrowdFormation
{
    IReadOnlyList<Vector3> GeneratePositions(int count);
}

И отдельная реализация:

public class EllipseCrowdFormation : ICrowdFormation
{
    private readonly CrowdFormationSettings settings;

    public EllipseCrowdFormation(CrowdFormationSettings settings)
    {
        this.settings = settings;
    }
}

Теперь SlimeCrowdManager не знает, как именно рассчитывается форма толпы. Он получает ICrowdFormation через Zenject и использует только метод GeneratePositions. 

[Inject]
private void Construct(ICrowdFormation crowdFormation)
{
    this.crowdFormation = crowdFormation;
}

5. ScriptableObject для настроек

Для параметров формации был создан CrowdFormationSettings : ScriptableObject. В него вынесены значения, влияющие на форму толпы:

  • baseSpacing
  • ellipseWidth
  • ellipseDepth
  • frontOffsetZ
  • randomOffsetAmount

Использование ScriptableObject удобно для Unity, потому что настройки можно менять через Inspector без изменения кода. Также можно создавать разные assets с настройками для разных уровней.

6. Dependency Injection в UI

Компонент CrowdCountLabel получает зависимость через Zenject: 

[Inject]
private void Construct(ISlimeCrowd slimeCrowd)
{
    this.slimeCrowd = slimeCrowd;
}

Благодаря этому UI не зависит напрямую от SlimeCrowdManager. Он работает только с интерфейсом ISlimeCrowd, то есть только читает состояние.

7. ProjectContext и глобальные сервисы

По заданию необходимо создать ProjectContext и вынести в него глобальные сервисы. Важно не переносить туда всё подряд. ProjectContext должен содержать только те зависимости, которые живут на протяжении всей игры и не относятся к конкретному уровню.

Для SlimeRush глобальными сервисами могут быть:

  • IInputService - единая обработка ввода;
  • ISaveSystem - сохранение и загрузка прогресса;
  • ISceneService - централизованная загрузка сцен;
  • IAudioService - глобальное управление звуком;
  • IGameProgressService - прогресс игрока между уровнями.

Не следует выносить в ProjectContext объекты конкретного уровня: SlimeCrowdManager, ворота, врагов, level spawner или настройки формации конкретной сцены. Эти зависимости должны оставаться в SceneContext.

8. Предлагаемый ProjectInstaller

Для полного соответствия заданию можно создать отдельный ProjectInstaller: 

public class ProjectInstaller : MonoInstaller
{
    public override void InstallBindings()
    {
        Container.Bind<IInputService>().To<InputService>().AsSingle();
        Container.Bind<ISaveSystem>().To<SaveSystem>().AsSingle();
        Container.Bind<ISceneService>().To<SceneService>().AsSingle();
        Container.Bind<IAudioService>().To<AudioService>().AsSingle();
    }
}

Этот installer должен быть подключён к объекту ProjectContext. После этого сервисы будут доступны во всех сценах проекта.

9. Выбор Lifetime

Для глобальных сервисов лучше использовать AsSingle, потому что они должны существовать в одном экземпляре на протяжении всей игры.

  • InputService - AsSingle: один источник ввода для всей игры.
  • SaveSystem - AsSingle: единая система сохранений.
  • SceneService - AsSingle: единая точка загрузки сцен.
  • AudioService - AsSingle: один глобальный контроллер звука.

AsTransient не подходит для таких сервисов, потому что каждый запрос создавал бы новый объект. Это может привести к потере состояния. AsCached можно использовать для ленивого создания объекта при первом запросе, но для базовых глобальных сервисов AsSingle проще и понятнее.

10. Преимущества Dependency Injection

Использование DI в проекте даёт следующие преимущества:

  • классы зависят от интерфейсов, а не от конкретных реализаций;
  • реализации можно заменять без изменения потребителей;
  • UI не получает лишний доступ к игровой логике;
  • уменьшается количество жёстких ссылок между MonoBehaviour;
  • зависимости класса видны явно через Construct или constructor;
  • код становится проще тестировать и расширять;
  • архитектура становится ближе к принципам чистого кода.

11. Что нужно приложить к сдаче

Для сдачи задания необходимо подготовить:

  • Unity-проект с интегрированным Zenject;
  • созданный и подключённый ProjectContext;
  • ProjectInstaller с глобальными сервисами;
  • пример использования хотя бы одного сервиса через [Inject];
  • скриншоты настройки ProjectContext и ProjectInstaller;
  • PDF-отчёт с описанием архитектурных решений.

12. Вывод

В проекте SlimeRush уже используется Zenject для управления зависимостями на уровне сцены. Были выделены интерфейсы для чтения и изменения состояния толпы, а расчёт формации был вынесен в отдельную зависимость ICrowdFormation.

Следующий шаг для полного выполнения задания - добавить ProjectContext и зарегистрировать в нём глобальные сервисы проекта. При этом игровые объекты конкретного уровня должны оставаться в SceneContext, так как они относятся к конкретной игровой сцене.
